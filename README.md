## 📋 Техническое задание

### Технические требования

- Python 3.8+
- Django 3+
- DRF 3.10+
- PostgreSQL 10+

При выполнении тестового задания можно дополнительно использовать любые сторонние Python-библиотеки без ограничений.

### Задание

Создать веб-приложение с API-интерфейсом и админ-панелью. Создать базу данных, используя миграции Django.

**Требования к реализации:**

1. Реализовать модель сети по продаже электроники. Сеть должна представлять собой иерархическую структуру из трёх уровней:
   - завод;
   - розничная сеть;
   - индивидуальный предприниматель.

2. Каждое звено сети ссылается только на одного поставщика оборудования (не обязательно предыдущего по иерархии). Уровень иерархии определяется не названием звена, а отношением к остальным элементам сети, т. е. завод всегда находится на уровне 0, а если розничная сеть относится напрямую к заводу, минуя остальные звенья, её уровень — 1.

3. Каждое звено сети должно обладать следующими элементами:
   - Название.
   - Контакты: email, страна, город, улица, номер дома.
   - Продукты: название, модель, дата выхода продукта на рынок.
   - Поставщик (предыдущий по иерархии объект сети).
   - Задолженность перед поставщиком в денежном выражении с точностью до копеек.
   - Время создания (заполняется автоматически при создании).

4. Сделать вывод в админ-панели созданных объектов. На странице объекта сети добавить:
   - ссылку на «Поставщика»;
   - фильтр по названию города;
   - admin action, очищающий задолженность перед поставщиком у выбранных объектов.

5. Используя DRF, создать набор представлений:
   - CRUD для модели поставщика (запретить обновление через API поля «Задолженность перед поставщиком»).
   - Добавить возможность фильтрации объектов по определённой стране.
   - Настроить права доступа к API так, чтобы только активные сотрудники имели доступ к API.

## ✅ Реализация

Все требования задания выполнены в полном объёме. Ниже — как каждый пункт реализован в проекте.

### Модель сети — единая таблица `NetworkNode`

Вместо трёх отдельных таблиц используется **единая модель `NetworkNode`** с полем `node_type` (тип звена: `FACTORY`, `RETAIL_NETWORK`, `INDIVIDUAL_ENTREPRENEUR`). Это решение:
- Упрощает запросы к базе данных (без JOIN между таблицами)
- Позволяет гибко фильтровать и сортировать по любым полям
- Облегчает добавление новых типов звеньев в будущем

### Иерархия и уровень звена

При сохранении узла поле `hierarchy_level` рассчитывается **автоматически** — алгоритм проходит по цепочке поставщиков до корневого узла (завода) и считает количество переходов. Защита от бесконечных циклов реализована через множество посещённых узлов.

```
Завод (уровень 0)
  └── Розничная сеть (уровень 1)
        └── ИП (уровень 2)
              └── ИП (уровень 3)
```

### Элементы каждого звена сети

| Требование | Реализация |
|-----------|-----------|
| Название | `NetworkNode.name` — `CharField(255)` |
| Контакты | Абстрактная модель `ContactInfoModel` с полями `email`, `country`, `city`, `street`, `house_number` |
| Продукты | Отдельная модель `Product` с `ForeignKey` на `NetworkNode` (поля: `name`, `model`, `release_date`) |
| Поставщик | `NetworkNode.supplier` — `ForeignKey("self")`, ссылка на другой узел сети |
| Задолженность | `NetworkNode.debt` — `DecimalField(max_digits=12, decimal_places=2)` с точностью до копеек |
| Время создания | Абстрактная модель `TimestampedModel` с полем `created_at` (`auto_now_add=True`) |

### Админ-панель

| Требование | Реализация |
|-----------|-----------|
| Вывод объектов | `NetworkNodeAdmin` с `list_display` — название, тип, уровень, поставщик, задолженность, город, страна, дата создания |
| Ссылка на поставщика | Метод `link_to_supplier` — кликабельная HTML-ссылка через `format_html`, ведёт на страницу редактирования поставщика |
| Фильтр по городу | `list_filter = ["city", ...]` — встроенный фильтр Django Admin по названию города |
| Admin action | `clear_debt` — обнуляет задолженность (`debt = 0.00`) у всех выбранных объектов через `queryset.update()` |

Дополнительно реализовано:
- **Инлайн-продукты** (`ProductInline`) — редактирование продуктов прямо на странице звена
- **Группировка полей** в `fieldsets` — логические блоки (основное, поставщик, контакты, системные)
- **Кастомные стили** — собственный CSS для улучшенного отображения
- **Поиск** — по названию, городу, стране

### API (DRF)

| Требование | Реализация |
|-----------|-----------|
| CRUD для модели | `NetworkNodeViewSet` (ModelViewSet) + `ProductViewSet` — полный набор CRUD-операций |
| Запрет обновления `debt` | Поле `debt` включено в `read_only_fields` сериализатора — невозможно изменить через API |
| Фильтрация по стране | `NetworkNodeFilter` с `django-filter` — параметр `?country=Россия` (регистронезависимый, `iexact`) |
| Права доступа | Кастомный permission `IsActiveEmployee` — проверяет `is_authenticated` и `is_active` |

## 📖 Как работать с API

### Способы авторизации

API поддерживает два способа авторизации:

| Способ | Когда использовать |
|--------|-------------------|
| **Token Authentication** | Для программных запросов (curl, Postman, код приложения) |
| **Session Authentication** | Для работы через браузер (DRF Browsable API) |

---

### Способ 1. Работа через браузер (Browsable API)

Самый простой способ — работа через браузер. Django REST Framework предоставляет удобный веб-интерфейс для всех API-эндпоинтов.

**Шаг 1.** Откройте `/api/` в браузере.

**Шаг 2.** Нажмите кнопку **«Log in»** в правом верхнем углу.

**Шаг 3.** Введите логин и пароль (в dev-окружении: `admin` / `admin`).

**Шаг 4.** После входа все страницы API доступны — можно просматривать данные, создавать, редактировать и удалять записи через HTML-формы.

> **Примечание:** При работе через браузер токен не нужен — используется сессионная авторизация (cookies).

---

### Способ 2. Работа через токен (curl, Postman, код)

Для программных запросов необходимо сначала получить токен, а затем передавать его в заголовке каждого запроса.

#### Шаг 1. Получение токена

```bash
curl -X POST http://localhost:5000/api/token/ \
  -H "Content-Type: application/json" \
  -d '{"username": "admin", "password": "admin"}'
```

Ответ:

```json
{"token": "a1b2c3d4e5f6..."}
```

> **Важно:** Сохраните значение токена — оно понадобится для всех последующих запросов.

#### Шаг 2. Использование токена

Токен нужно передавать в HTTP-заголовке `Authorization` в формате `Token <значение>`:

```bash
curl http://localhost:5000/api/network-nodes/ \
  -H "Authorization: Token a1b2c3d4e5f6..."
```

#### Шаг 3. Примеры запросов

**Получить список всех звеньев сети:**

```bash
curl http://localhost:5000/api/network-nodes/ \
  -H "Authorization: Token a1b2c3d4e5f6..."
```

**Создать завод (уровень 0):**

```bash
curl -X POST http://localhost:5000/api/network-nodes/ \
  -H "Authorization: Token a1b2c3d4e5f6..." \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Электроника Плюс",
    "node_type": "FACTORY",
    "email": "factory@example.com",
    "country": "Россия",
    "city": "Москва",
    "street": "Ленина",
    "house_number": "1"
  }'
```

**Создать розничную сеть (привязана к заводу с id=1):**

```bash
curl -X POST http://localhost:5000/api/network-nodes/ \
  -H "Authorization: Token a1b2c3d4e5f6..." \
  -H "Content-Type: application/json" \
  -d '{
    "name": "ТехноМаг",
    "node_type": "RETAIL_NETWORK",
    "supplier": 1,
    "email": "retail@example.com",
    "country": "Россия",
    "city": "Санкт-Петербург",
    "street": "Невский проспект",
    "house_number": "25"
  }'
```

**Обновить звено (частичное обновление):**

```bash
curl -X PATCH http://localhost:5000/api/network-nodes/2/ \
  -H "Authorization: Token a1b2c3d4e5f6..." \
  -H "Content-Type: application/json" \
  -d '{"name": "ТехноМаг Обновлённый"}'
```

**Удалить звено:**

```bash
curl -X DELETE http://localhost:5000/api/network-nodes/2/ \
  -H "Authorization: Token a1b2c3d4e5f6..."
```

**Фильтрация по стране:**

```bash
curl "http://localhost:5000/api/network-nodes/?country=Россия" \
  -H "Authorization: Token a1b2c3d4e5f6..."
```

**Создать продукт:**

```bash
curl -X POST http://localhost:5000/api/products/ \
  -H "Authorization: Token a1b2c3d4e5f6..." \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Смартфон X100",
    "model": "X100 Pro",
    "release_date": "2025-01-15",
    "network_node": 1
  }'
```

### Способ 3. Проверка токена через API Root

На главной странице `/api/` есть форма для проверки токена. Вставьте **только значение токена** (без кавычек, без скобок, без `"token":`) в поле и нажмите POST. Если токен действителен, отобразятся ссылки на все ресурсы.

> **Примечание:** В ответе токен отображается в замаскированном виде (например, `a1b2****...****e5f6`) для безопасности.

---

### Важные особенности API

| Особенность | Описание |
|------------|----------|
| **Защита поля `debt`** | Задолженность перед поставщиком нельзя изменить через API — поле доступно только для чтения |
| **Автоматический расчёт уровня** | Поле `hierarchy_level` рассчитывается автоматически при сохранении |
| **Только активные сотрудники** | Доступ к API имеют только пользователи с флагом `is_active=True` |
| **Типы узлов** | `FACTORY` (завод), `RETAIL_NETWORK` (розничная сеть), `INDIVIDUAL_ENTREPRENEUR` (ИП) |
| **Маскировка токена** | В ответах API токен никогда не показывается полностью |

---

### Коды ответов

| Код | Значение |
|-----|---------|
| `200` | Успешный запрос |
| `201` | Объект успешно создан |
| `204` | Объект успешно удалён |
| `400` | Ошибка валидации (проверьте отправленные данные) |
| `401` | Не авторизован (неверный или отсутствующий токен) |
| `403` | Доступ запрещён (пользователь неактивен) |
| `404` | Объект не найден |

- **API** — CRUD операции, защита `debt`, фильтрация по стране, права доступа
- **Сериализаторы** — сериализация, десериализация, read-only поля
- **Админка** — отображение, ссылка на поставщика, admin action очистки задолженности
- **Права доступа** — `IsActiveEmployee`, анонимные пользователи, неактивные
- **Pydantic-схемы** — валидация типов узлов, email, ограничения полей
- **Management-команды** — `create_superuser`, `load_demo_data`
- **Утилиты** — форматирование валюты, отображение типов узлов
- **Конфигурация** — settings, URLs, WSGI/ASGI, fallback базы данных
- **Токен-аутентификация** — получение и использование токена

## 🚀 Запуск проекта

### Переменные окружения

| Переменная | Обязательная | Описание |
|------------|:---:|----------|
| `SESSION_SECRET` | Да | Секретный ключ Django (`SECRET_KEY`) |
| `DATABASE_URL` | Нет* | URL подключения к PostgreSQL |
| `DEBUG` | Нет | Режим отладки (`True`/`False`, по умолчанию `True`) |
| `ALLOWED_HOSTS` | Нет | Разрешённые хосты (по умолчанию `*`) |
| `DB_ENGINE` | Нет* | Движок БД (если нет `DATABASE_URL`) |
| `DB_NAME` | Нет* | Имя БД (если нет `DATABASE_URL`) |
| `DB_USER` | Нет* | Пользователь БД (если нет `DATABASE_URL`) |
| `DB_PASSWORD` | Нет* | Пароль БД (если нет `DATABASE_URL`) |
| `DB_HOST` | Нет* | Хост БД (если нет `DATABASE_URL`) |
| `DB_PORT` | Нет* | Порт БД (если нет `DATABASE_URL`) |

\* Используется либо `DATABASE_URL`, либо набор `DB_*` переменных.

### Команды

```bash
# Миграции
python manage.py migrate

# Создание суперпользователя
python manage.py create_superuser

# Загрузка демо-данных
python manage.py load_demo_data

# Запуск сервера
python manage.py runserver 0.0.0.0:5000
```
